// The Problem
Financial data deserves better than CSV in a ZIP

Traditional workflows with CSV and Parquet leave critical gaps that cause silent failures in backtesting pipelines.

✗
No Standardized Metadata
Every vendor uses different column names. You're guessing what "Vol" means and hoping the timezone is UTC.
✗
No Quality Guarantees
Ghost spikes, negative prices, duplicate timestamps — you discover them 3 months later when your backtest explodes.
✗
No Audit Trail
Was this data validated? When? By whom? There's no record. You just have to trust it.
✗
No SQL Without a Database
Want to filter by volume? Load the entire file into Pandas first, or spin up a database server.
// The Solution
One file. Everything included.
✓
Smart Metadata
Ticker, exchange, timezone, timescale, asset class — all validated by Pydantic and embedded in the Arrow schema.
✓
5 Automatic Audit Checks
OHLC consistency, positive prices, chronological order, outlier detection (5σ), gap identification — all before sealing.
✓
Sealed Audit Trail
Every file records: passed/failed, timestamp, gap count, outlier count, auditor version. Fully reproducible.
✓
SQL Over Files
DuckDB reads .qcr files directly — no loading, no copying, no database server. Just SQL.

// Encoding Strategy
Why every byte matters

Each column uses a Parquet encoding chosen for its data characteristics — not generic defaults.

timestamp
DELTA_BINARY_PACKED
Timestamps are monotonically increasing integers. Delta encoding stores only the tiny diffs between consecutive values — often just a single repeated interval.
open / high / low / close
BYTE_STREAM_SPLIT
Splits IEEE-754 float bytes into four homogeneous streams. The exponent bytes cluster together, the mantissa bytes cluster together — ZSTD compresses each stream dramatically better.
volume
DELTA_BINARY_PACKED
Integer volumes with gradual changes benefit from delta encoding. Consecutive volumes often differ by small amounts, so deltas are tiny.

Pack data with metadata and automatic audit

Three lines to go from raw DataFrame to a sealed, validated .qcr file.

from qcr import save_sealed_qcr, run_audit
from qcr.schema import QcrMetadata, AssetClass, Timescale

metadata = QcrMetadata(
    ticker="AAPL",
    asset_class=AssetClass.EQUITY,
    currency="USD",
    exchange="XNAS",
    timezone="America/New_York",
    timescale=Timescale.ONE_DAY,
)

audit = run_audit(df, Timescale.ONE_DAY)
save_sealed_qcr(df, metadata, audit.to_audit_trail(), "AAPL.qcr")

The Financial Data Linter

Inject bad data and watch the auditor catch every issue before it reaches your backtest.

from qcr.auditor import run_audit, Severity
from qcr.schema import Timescale

# Inject problems into the data
df[100, "close"] = -5.0       # Negative price!
df[50, "timestamp"] = df[49]  # Duplicate timestamp!
df[200, "high"] = 1.0         # High < Low!

result = run_audit(df, Timescale.ONE_MINUTE)
for issue in result.issues:
    print(f"[{issue.severity}] {issue.check}: {issue.message}")

SQL Over Files — No Database Required

DuckDB reads .qcr files directly. Filter, aggregate, JOIN — all with standard SQL.

from qcr.vtable import query_qcr, query_qcr_df

result = query_qcr("""
    SELECT timestamp, close, volume
    FROM 'AAPL.qcr'
    WHERE close > 150 AND volume > 1000000
    ORDER BY volume DESC
    LIMIT 5
""")

# Cross-file JOIN
result = query_qcr("""
    SELECT a.timestamp, a.close AS aapl, b.close AS msft
    FROM 'AAPL.qcr' a
    JOIN 'MSFT.qcr' b ON a.timestamp = b.timestamp
""")

CLI — One Command to Rule Them All

Pack, inspect, ingest, and query from the terminal.

# Pack a CSV into a sealed .qcr file
$ qcr pack aapl.csv --ticker AAPL --timescale 1d

# Inspect metadata (reads only the Parquet footer)
$ qcr info AAPL.qcr

# Fetch live data from Yahoo Finance
$ qcr ingest AAPL --start 2024-01-01 --end 2024-12-31

# Query with SQL
$ qcr sql "SELECT * FROM 'AAPL.qcr' WHERE close > 150" --format table

Logical Consistency Check

Validates the fundamental OHLC invariants that must hold for every candle:

high >= open   high >= close   high >= low
low <= open   low <= close

Uses vectorized Polars filters — checks all rows simultaneously. Any violation is an ERROR that blocks sealing, with exact row indices reported for debugging.

Positive Price Check

All four price columns (open, high, low, close) must be strictly greater than zero. Negative or zero prices indicate data corruption — common in vendor feeds after corporate actions or system errors.

Severity: ERROR — blocks sealing.

Chronological Integrity Check

Computes timestamp.diff() across the entire series using Polars. Any row where the diff is ≤ 0 (duplicate or backward timestamp) is flagged.

This catches duplicate rows from vendor re-deliveries and timezone conversion bugs. Severity: ERROR.

Outlier Detection (5σ)

Computes log close-to-close returns: ln(close[i] / close[i-1]). Calculates the global mean and standard deviation, then flags any return where |z-score| > 5.0.

These may be legitimate (earnings gaps, circuit breakers) so the severity is WARNING — recorded in the audit trail but doesn't block sealing.

Gap Detection

Compares consecutive timestamp diffs against the expected interval for the declared Timescale. For example, 1-minute data should have ~60s between candles.

Tick data is skipped (no fixed interval). Monthly uses a ~30-day approximation. Severity: WARNING.
